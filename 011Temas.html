    Adaptar el tema

    Como adecuar la disposición y presentación de las páginas.

    Escogiendo un tema

    Ya hemos visto que el formato de la página, disposición y comportamiento de los distintos elementos, y aspecto en general se consigue aplicando un tema. MkDocs proporciona un par de temas internos (built-in) así como la posibilidad de utilizar los creados por otras personas. Todos ellos pueden ser configurados y adaptados a nuestro gusto. También podemos crear el nuestro propio.

    Para utilizar uno de los temas estándar, basta con añadir al fichero de configuración mkdocs.yml la siguiente entrada:

    theme: mkdocs

    o bien:

    theme: readthedocs

    El primero es el que se usa por defecto.

    mkdocs

    El tema “MkDocs” solo admite dos niveles en la barra de navegación, es decir, un menú horizontal y pulsando en cada opción, menús desplegables.

    Además de las opciones habituales, el tema mkdocs incluye parámetros que podemos incluir en el archivo de configuración. Por ejemplo, la función de resaltado de código fuente se puede configurar:

    • highlightjs: resalta el texto fuente usando la biblioteca highlight.js de JavaScript. Por defecto es True.

    • hljs_style: la librería highlight.js proporciona 79 diferentes estilos (variaciones de color) para resaltar los elementos. Poner el nombre del estilo deseado. Por defecto es github.

    • hljs_languages: por defecto, highlight.js solo admite 23 lenguajes. Incluir aquí una lista adicional de lenguajes para añadir soporte para los mismos. Por ejemplo:

      theme:
    name: mkdocs
    highlightjs: true
    hljs_languages:
        - yaml
        - rust

    Otra posibilidad del tema MkDocs es definir atajos de teclado con la opción shortcuts:

        theme:
        name: mkdocs
        shortcuts:
            help: 191    # ?
            next: 78     # n
            previous: 80 # p
            search: 83   # s

    Todos los valores han de ser códigos numéricos. Conviene usar códigos presentes en cualquier teclado. Para determinar los valores, mirar en https://keycode.info/. Los posibles atajos son:

    • help: Muestra una ventana con la lista de atajos de teclado. Por defecto el valor es 191 (tecla ?)

    • next: Pasa a la siguiente página. Por defecto: 78 (tecla n)

    • previous: Página anterior. Por defecto: 80 (tecla p)

    • search: Muestra ventana de búsqueda. Por defecto: 83 (tecla s)

    readthedocs

    Es un clon del tema por defecto del servicio Read the Docs.

    Además de las opciones por defecto, este tema cuenta con:

    • highlightjs: Permite el resaltado de bloques de código fuente usando la librería highlight.js de JavaScript. Por defecto es True.

    • hljs_languages: Por defecto, highlight.js solo admite 23 lenguajes. Incluir aquí una lista adicional de lenguajes para añadir soporte para los mismos:

      theme:
    name: mkdocs
    highlightjs: true
    hljs_languages:
        - yaml
        - rust

    • include_homepage_in_sidebar: si el valor es true (por defecto) añade la página de inicio a la barra lateral aunque debemos incluirla en la la lista nav del archivo de configuración. En todo caso, pulsando con el ratón sobre el nombre del sitio o en el logo, siempre nos llevará a la página de inicio.

    • prev_next_buttons_location: Puede ser bottom, top, both , o none. Muestra los botones [Next] y [Previous] de acuerdo a este parámetro, que por defecto es bottom.

    • navigation_depth: Máxima profundidad de navegación en el número de niveles a mostrar en la barra lateral. Valor por defecto: 4.

    • collapse_navigation: Solo incluye las cabeceras de sección en la barra lateral. Por defecto es True.

    • titles_only: Solo incluye títulos de página en la barra lateral, excluyendo las cabeceras de sección. Por defecto es False.

    • sticky_navigation: Si es true (por defecto), la barra lateral se desplaza arriba/abajo en la medida que desplazamos el contenido de la página.

    Temas de terceros

    Podemos encontrar una lista de temas de terceros en la wiki de MkDocs. Por ejemplo, vamos a instalar el tema Material:

    pip3 install mkdocs-material

    Recuérdese que para hacer esta instalación necesitaremos permisos de administrador. Para usar esos temas, en este caso Material, escribimos en el archivo de configuración mkdocs.yml la siguiente entrada:

    theme: material

    Y al volver a compilar las páginas debería hacer el efecto deseado. Para configurar el nuevo tema, tendremos que acudir a la documentación del mismo.

    Añadiendo hojas de estilos

    Podemos usar las opciones de configuración extra_css y extra_javascript para añadir nuevos archivos de estilo y scripts. Por ejemplo, para cambiar el color de las cabeceras, en el fichero de configuración escribimos:

    extra_css:
    - css/estilos.css
    - css/mas_estilos.css

    Y en la carpeta de documentos, creamos un archivo css/estilos.css con lo siguiente:

    h1 {
  color: red;
}

    Adaptando el tema

    Para hacer modificaciones en un tema, comenzamos por acudir al repositorio del mismo y examinar la lista de archivos. Por ejemplo, el tema mkdocs tiene una estructura que es más o menos como sigue:

    - css\
- fonts\
- img\
  - favicon.ico
  - grid.png
- js\
- 404.html
- base.html
- content.html
- nav-sub.html
- nav.html
- toc.html

    Para modificar un archivo, tenemos que copiarlo a nuestro proyecto, ubicándolo en una carpeta de igual nombre que en el tema original. MkDocs tomará con prioridad los archivos locales.

    Las carpetas con archivos del tema deben situarse en una carpeta que “cuelgue” de la principal del proyecto, en paralelo a la carpeta de documentos. La indicamos en el archivo de configuración:

    theme:
    name: mkdocs
    custom_dir: tema/

    Con esto le estamos diciendo que use el tema mkdocs, pero que acuda a nuestra carpeta /tema/ para ver si hay archivos personalizados, que tienen preferencia sobre los originales.

    Por ejemplo, el tema original tiene un archivo 404.html que muestra la página “file not found cuando una página enlazada no existe. Podemos incluir nuestro propio archivo 404.html* en la carpeta a puntada por el parámetro theme.custom_dir.

    Nuestro proyecto podría tener la siguiente estructura:

    - config.yml
- docs/
  - index.md
  - otrapagina.md
- tema/
  - img/
    - favicon.ico
  - 404.html
- css
  - estilos.css

    Los archivos incluidos en la carpeta /tema/ reemplazarán los originales con el mismo nombre o se añadirán al tema.

    Podríamos copiar en esa carpeta todo el tema original completo, y en el archivo de configuración escribir:

    theme:
    name: null
    custom_dir: tema/

    Si no definimos un tema ni una carpeta para alojar el mismo, MkDocs usará el tema por defecto.

    Anatomía de un tema

    Lo mínimo es un archivo main.html, que es una plantilla con marcas Jinja2 (el lenguaje de marcas utilizado por MkDocs).

    Por ejemplo si nuestro proyecto es:

    mkdocs.yml
docs/
    index.md
    acercade.md
tema/
    main.html
    ...

    … escribiremos lo siguiente en mkdocs.yml:

    theme:
    name: null
    custom_dir: 'tema/'

    … y estaremos usando como tema los archivos y plantillas de la carpeta /tema/. Pero si escribiéramos:

    theme:
    name: otro_tema
    custom_dir: 'tema/'

    lo que estaríamos haciendo es usar otro tema reescribiendo main.html.

    El fichero main.html más simple podría ser:

    <!DOCTYPE html>
<html>
  <head>
    <title>{% if page.title %}{{ page.title }} - {% endif %}
    {{ config.site_name }}</title>
  </head>
  <body>
    {{ page.content }}
  </body>
</html>

    El texto de cada página se insertará en lugar de la marca {{page.content}}. Podemos añadir etiquetas para enlazar con hojas de estilo y scripts como en cualquier archivo html.

    Variables

    El texto de una plantilla puede incluir variables, que pueden ser a nivel global o de página. Una variable se escribe como

    {{ nombre_variable }}

    Las siguientes variables están disponibles “de serie” para cualquier plantilla:

    • config. Incluye las siguientes opciones:

      • config.site_name
      • config.site_url
      • config.site_author
      • config.site_description
      • config.extra_javascript
      • config.extra_css
      • config.repo_url
      • config.repo_name
      • config.copyright
      • config.google_analytics

    • nav

      La variable nav es una lista de páginas definida en el archivo de configuración. Además incluye los siguientes atributos:

      • nav.homepage: página principal del sitio

      • nav.pages: lista de páginas a incluir en la barra de navegación. Puede haber otras páginas no incluidas en la barra. Afecta al comportamiento de los enlaces “next page” y “previous page”.

    • base_url

      La variable base_url proporciona un path a la carpeta principal del proyecto. Puede usarse directamente precediendo una URL relativa, pero es mejor usar el filtro url para asegurarnos de una correcta transformación:

      {{ base.url|url }}{{ page.url|url }}{{ page.title }}

      En el lenguaje de marcas Jinja2, los filtros consisten en aplicar una conversión a una variable. Se coloca un símbolo “|” entre el nombre de la variable y el filtro a aplicar.

    • mkdocs_version

      Versión actual de MkDocs.

    • build_date_utc

      Objeto de la clase datetime de Python que representa la fecha y hora de compilación en formato UTC. Resulta útil para saber si los documentos son recientes.

    • pages

      Una lista con todas las páginas del proyecto ordenadas alfabéticamente por directorio y nombre de archivo. Contiene páginas no incluidas en la variable nav. El orden de las páginas no tiene por que coincidir con el de la barra de navegación.

    • page

      En las plantillas no utilizadas por un archivo Markdown, la variable page tiene un valor None. En caso contrario, la variable tiene los siguientes atributos:

      • page.title

        Título de la página actual.

      • page.content

        HTML generado a partir de Markdown, que se mostrará como documentación.

      • page.toc

        Una lista de enlaces que se usa para formar la tabla de contenidos. Cada elemento tiene los siguientes atributos:

        • title: texto del elemento.
        • url: URL para apuntar a un objeto
        • level: Nivel en la tabla de contenidos, con base cero.
        • children: lista de elementos anidados.

      • page.meta

        Es una colección con los metadatos incluidos en el front matter de la página. Por ejemplo:

        source: generics.py
        mixins.py

# Título de la página

Contenido...

      • page.url

        La URL de la página relativa a la apuntada por site_dir. Es preferible usar el filtro url para asegurarse de que la URL es relativa a la página actual.

        <a href="{{ page.url|url }}">{{ page.title }}</a>

      • page.abs_url

        La URL absoluta de la página de acuerdo al parámetro de configuración site_url. Incluye los subdirectorios incluidos en site_url, pero no el dominio. Lo apuntado por base_url no debería usarse con esta variable.

        Por ejemplo, si tenemos:

        site_url: https://ejemplo.com/

        el valor de page.abs_url para la página foo.md será /foo/.

        Pero si tenemos:

        site_url: https://ejemplo.com/bar/

        entonces el valor de page.abs_url para foo.md será /bar/foo/.

      • page.canonical_url

        Dirección URL completa tal y como está determinado por el valor asignado a site_url en la configuración. El valor incluye el dominio y cualquier subdirectorio incluido en site_url. base_url no debería utilizarse con esta variable.

      • page.edit_url

        La URL completa al texto fuente en el repositorio. Generalmente se usará para proporcionar un enlace para editar el texto fuente. El valor de base_urlno debería usarse con esta variable.

      • page.is_homepage

        True en el caso de la página principal del sitio, y False en el resto de páginas. Puede usarse en conjunción con otros atributos de page para alterar el comportamiento. Por ejemplo, para mostrar un título diferente en la página principal, en la plantilla escribimos:

        {% if not page.is_homepage %}{{ page.title }} - {% endif %}{{ site_name }}

      • page.previous_page

        El nombre de la página anterior. Si estamos en la primera página, el valor será None. Sucede lo mismo si estamos en una página no incluida en la barra de navegación.

      • page.next_page

        Similar al anterior, pero apunta a la página siguiente.

      • page.parent

        Sección inmediatamente situada a nivel superior en la barra de navegación. El valor es none si estamos en el nivel superior.

      • page.children

        Las páginas no tienen páginas hijas. El valor siempre es none. Esta propiedad tiene sentido para el caso de las secciones, donde devuelve una lista.

      • page.active

        Si el valor es true, indica que esta es la página actual. Por defecto es false.

      • page.is_section

        Indica que en la barra de navegación es un objeto “section”. El valor es false en el caso de las páginas páginas.

      • page.is_page

        Indica que un objeto en la barra de navegación es una página. El valor es true para todas las páginas.

      • page.is_link

        Indica que el objeto es un enlace en la barra de navegación. Siempre es false para las páginas.

    Veamos un ejemplo de uso en una plantilla, usando el lenguaje de marcas Jinja:

    <ul>
    {% for toc_item in page.toc %}
        <li><a href="{{ toc_item.url }}">{{ toc_item.title }}</a></li>
           {% for toc_item in toc_item.children %}
               <li><a href="{{ toc_item.url }}">{{ toc_item.title }}  </a></li>
           {% endfor %}
    {% endfor %}
</ul>

    La variable nav es una lista de objetos del tipo section, page o link. Los primeros pueden tener otros anidados.

    Los objetos section tienen los siguientes atributos:

    • section.title

      Título de la sección.

    • section.parent

      Sección padre. Si estamos al nivel superior, None.

    • section.children

      Una lista con todos los objetos incluidos en la sección, que pueden ser a su vez secciones, páginas o enlaces.

    • section.active

      Si es true, indica que una página hija de esta sección es la página actual. Puede usarse para resaltar el nombre de la sección. Por defecto es false.

    • section.is_section

      Indica que el objeto es una sección.

    • section.is_page

      Indica que el objeto es una página. El valor es false para las secciones.

    • section.is_link

      Indica que el objeto es un enlace.

    Los objetos link pueden tener los siguientes atributos

    • link.title

      Título del enlace.

    • link.url

      La URL apuntada por el enlace. Debería ser una URL absoluta, no precedida por base_url.

    • link.parent

      Elemento padre del enlace. None si estamos al nivel superior.

    • link.children

      Los enlaces no tienen elementos hijos. Siempre es None.

    • link.active

      Los enlaces nunca pueden ser páginas activas. Siempre es False.

    • link.is_section

      Siempre false para enlaces.

    • link.is_page

      Siempre false para enlaces.

    • link.is_link

      Siempre true para enlaces.

    Bloques

    En lugar de sobreescribir un archivo de un tema, podemos hacerlo con parte del archivo. Las plantillas se pueden dividir en bloques de texto, delimitados de la siguiente forma:

    {% block nombre %}
...contenido html...
{% endblock %}

    Para sobreescribir un bloque de un archivo, en la carpeta apuntada por custom_dir creamos un fichero con el mismo nombre:

    {% extends "base.html" %}

{% block htmltitle %}
<title>Título personalizado</title>
{% endblock %}

    La plantilla principal main.html suele ser una ampliación de base.html, y el bloque htmltitle se suele utilizar para definir la cabecera de la página web.

    Obviamente, tendremos que examinar los archivos del tema para saber que bloques son sobreescribibles. Los termas MkDocs y ReadTheDocs proporcionan los siguientes bloques:

    • site_meta: etiquetas meta en la sección <head>.
    • htmltitle: título de la página.
    • styles: enlaces a hojas de estilo.
    • libs: librerías JavaScript (jQuery, etc).
    • scripts: scripts JavaScript.
    • analytics: script analytics.
    • extrahead: bloque vacío en la sección <head>, para añadir elementos a nuestro gusto.
    • site_name: nombre del sitio en la barra de navegación.
    • site_nav: enlaces de navegación.
    • search_box: recuadro para introducir búsquedas.
    • next_prev: botones next y previous.
    • repo: enlace al repositorio.
    • content: tabla de contenidos.
    • footer: pie de página.
