    Configurando el proyecto

    En este capítulo daremos un repaso al archivo de configuración y sus opciones.

    El archivo de configuración

    El proceso de conversión y generación del sitio web se configura en buena parte adaptando las definiciones de parámetros en el archivo mkdocs.yml, que es un fichero de texto en formato YAML ubicado en la carpeta principal.

    Todos los parámetros son opcionales, excepto site_name, que representa el título del proyecto. Por lo tanto, un archivo mkdocs.yml en su expresión mínima puede ser:

    site_name: Mi proyecto

    Formato YAML

    Los archivos YAML son ficheros de texto que se utilizan para almacenar datos. MkDocs y Jekyll lo utilizan para crear archivos de configuración con los parámetros del proceso de conversión.

    La sintaxis es sencilla y fue diseñada buscando que fuera muy legible. Se utiliza una sintaxis basada en el sangrado, que debe ser consistente y basado en espacios en blanco, no admitiéndose caracteres de tabulación.

    En principio, un archivo YAML contiene una colección de parejas clave:valor, cada una de ellas en línea aparte. Se admiten líneas en blanco.

    Cada variable puede ser de diferente tipo, que será deducido automáticamente:

    • un texto. Las comillas son opcionales, y pueden usarse apóstrofos en su lugar:

      nombre: "Luis Gutierrez"

    • un número:

      edad: 58

    • un valor booleano cuyo valor puede ser true o false

      activado: true

    • una lista. Los elementos se escriben en líneas aparte, precedidos por un guión

      participes:
    - Luis
    - Ana
    - Juan

    • una colección de elementos clave:valor

      opciones:
    idioma: es
    avisos: true

    Los parámetros pueden anidarse. Por ejemplo, podemos tener una lista donde uno o varios de sus elementos sean, a su vez, listas.

    El carácter # delimita lo que será considerado como comentarios e ignorado. Se toma como tal todo el texto desde # hasta el fin de línea

    #-------------------------
# opciones del proceso
#-------------------------

opciones:
    idioma: es
    avisos: true

    Podemos asignar a un parámetro un bloque de texto de varias líneas, si comienza con un carácter >:

    descripción: >
       Texto multilínea
       que será unido 
       en una sola línea
       de texto

       Una línea en blanco
       permite separar párrafos

    Aunque podemos preservar los saltos de línea si usamos | en lugar de >:

    descripción: |
       Texto multilínea
       formado por 
       cuatro líneas que
       serán respetadas

    Parámetros Mkdocs de proyecto

    • site_url

      Establece la dirección canónica del sitio web y genera una etiqueta en la sección header. El valor por defecto es null:

      <head>
<link rel="canonical" href="https://www.misitio.com/">
</head>

    • repo_url

      Muestra en cada página un enlace al repositorio indicado por esta variable (GitHub, Bitbucket, GitLab, …). El valor por defecto es null:

      repo_url: https://github.com/misitio

    • repo_name

      Determina el nombre del repositorio. Por defecto, lo intenta obtener a partir de repo_url.

    • edit_uri

      Tomando repo_url como dirección base del repositorio de documentos, edit_uri se añade para formar la dirección para editar la página. Esto facilita el acceso a la misma. Se admiten caracteres como (‘?’) y (‘#’).

      Valor por defecto: edit/master/docs/ para GitHub y GitLab, y src/default/docs/ para Bitbucket si repo_url apunta a estos dominios. En caso contrario, el valor por defecto será null. Por ejemplo:

      edit_uri: '?query=root/path/docs/'

edit_uri: '#root/path/docs/'

edit_uri: root/path/docs/

    Nota:

    En algunos repositorios como GitHub, Bitbucket o GitLab), el valor de edit_uri se deriva de ‘repo_url’ y no necesita establecerse manualmente.

    Por ejemplo, en GitHub edit_uri será edit/master/docs/. En Bitbucket será src/default/docs/.

    Para usar una URI diferente de la que se calcula por defecto (por ejemplo, otra rama de desarrollo), utilizar edit_uri. Asignarle una cadena vacía si no queremos un enlace “edit URL” mostrado en la página

    Atención:

    En GitHub y GitLab, el path por defecto (edit/master/docs/) abre el documento en el editor online. Esto requiere que nos hayamos conectado con nuestra cuenta de usuario y contraseña. En caso contrario, seremos redireccionados a una página de conexión. Alternativamente, podemos usar un path “blob” (blob/master/docs/) para abrir una vista de solo lectura, que admite acceso anónimo.

    • site_description

      Un texto descriptivo que se insertará en la sección <head> como metadato del archivo html. Por defecto es null

    • site_author

      El nombre del autor se insertará en la sección <head> como metadato. Por defecto es null

    • copyright

      Información de copyright a incluir en las páginas si el tema lo tiene contemplado. Por defecto es null

    • google_analytics

      Configura Google analytics. Por defecto es null. Por ejemplo:

      google_analytics: ['UA-36723568-3', 'mkdocs.org']

    • remote_branch

      Rama del repositorio cuando queremos subir los documentos a GitHub mediante el comando:

      mkdocs gh-deploy

      Este comando es una alternativa de mkdocs build, y además de generar el sitio web, automatiza la subida al repositorio. También podemos establecer la rama como parámetro en la línea de comandos. Por defecto es gh-pages

    • remote_name

      Nombre remoto donde subir los contenidos usando gh-deploy. Podemos establecerlo en la línea de comandos. Por defecto es origin.

    La barra de navegación

    El apartado nav se usa para determinar el formato de la navegación. Veamos, por ejemplo, una barra con tres entradas:

    nav:
    - 'Introducción': 'index.md'
    - 'Guía de usuario': 'user-guide.md'
    - 'Acerca de': 'about.md'

    Todos los paths deben ser relativos a la carpeta donde se ubica el archivo mkdocs.yml.

    Los elementos pueden incluir enlaces a sitios externos. En este caso, los títulos, que son opcionales para enlaces internos, son obligatorios para los externos.

    Un sitio externo puede ser una URL absoluta o relativa. Cualquier path no encontrado durante la conversión será asumido como enlace externo.

    Un ejemplo:

    nav:
    - Home: index.md
    - Guía de usuario: user-guide.md
    - Referencia: https://ejemplo.com/

    A veces un proyecto está alojado en un subdirectorio y queremos enlazar a otro. Podemos usar paths relativos:

    site_url: https://ejemplo.com/foo/

nav:
    - Home: ../
    - Guía de usuario: user-guide.md
    - Referencia: /ref/

    Véase que site_url indica que el proyecto está en la carpeta /foo/ dentro del dominio. Por lo tanto, Home tiene un enlace relativo que apunta a https://ejemplo.com/. La tercera entrada usa un path absoluto apuntando a https://ejemplo.com/ref/.

    Por defecto, si no especificamos nav, esta contendrá una lista de todos los documentos markdown apuntados por docs_dir, incluyendo subdirectorios, y ordenando las páginas alfabéticamente.

    Parámetros del tema utilizado

    La opción theme establece el tema a utilizar. Puede ser el nombre del tema o bien una lista de parámetros. En el primer caso:

    theme: mkdocs

    Este es el valor por defecto.

    Un ejemplo del segundo caso:

    theme:
    name: mkdocs
    custom_dir: miTema/
    static_templates:
        - mapa.html
    include_sidebar: false

    Los parámetros de un tema pueden ser:

    • name:

      Nombre del tema. Para los temas built-in puede ser mkdocs o readthedocs. Para temas de terceros, véase la documentación de los mismos.

    • custom_dir:

      Si queremos adaptar el tema a nuestro gusto, debemos reescribir algunos ficheros y dejarlos en la carpeta apuntada por custom-dir. Puede escribirse relativa a la carpeta principal (la que tiene el archivo de configuración) o bien con una referencia absolut.

    • static_templates:

      Una lista de plantillas, bien en la carpeta de plantillas del tema, o en la carpeta apuntada por custom_dir.

    • otros parámetros propios del tema utilizado.

    Directorios

    Además de la carpeta apuntada por custom-dir, con las plantillas del tema original adaptadas a nuestro gusto, tenemos otras carpetas que se pueden configurar:

    • Carpeta de documentos

      Por defecto los documentos se guardan en la carpeta docs. Podemos establecer otra mediante el parámetro docs_dir.

    • Carpeta del sitio web. Cuando compilemos el proyecto, se generará una carpeta sitecon los resultados. Podemos establecer otra carpeta mediante la opción site_dir.

    • Archivos css

      El parámetro extra_css es una lista de archivos .css, ubicados en la carpeta de documentos, que queremos incluir en la compilación. Por defecto es una lista vacía, que en lenguaje Python se representa mediante [].

      extra_css:
    - css/estilos.css
    - css/masEstilos.css

    • Archivos javascript

      El parámetro extra_javascript es una lista de archivos javascript, ubicados en la carpeta de documentos, que queremos incluir en la compilación. Por defecto es una lista vacía [].

    • Plantillas extra

      El parámetro extra_templates contiene una lista de plantillas ubicadas en nuestra carpeta de documentos. Por defecto es una lista vacía.

    Extra

    Este apartado es una lista de parámetros que podemos añadir a nuestro gusto cuando diseñemos temas. Por ejemplo, si queremos crear una variable version para mostrarla en las páginas:

    extra:
    version: 1.0

    Por defecto, extra es un conjunto vacío de claves.

    Parámetros de visualización

    • use_directory_urls

      Valores: trueo false. Controla el estilo utilizado para generar enlaces a páginas. Valor por defecto: true.

      El proceso de generación de páginas web convierte los enlaces markdown en enlaces html. Cuando el valor de este parámetro sea true, para cada página se creará un directorio y la página se llamará index.html. Los enlaces apuntarán a la carpeta, y no al archivo index.

      Cuando el valor sea false, las páginas generadas se llamarán igual que los archivos markdown (cambiando la extensión), y los enlaces apuntarán directamente a los archivos.

      Si queremos que los enlaces sean completos, con el nombre de la página enlazada, poner este parámetro como false. Resulta útil cuando queramos visualizar las páginas en nuestro entorno local, o en una plataforma que no redireccione automáticamente a index.html (GitHub si lo hace).

      Algunos ejemplos:

      Archivo a enlazar valor: true valor: false
      index.md /index.html /index.html
      guia_de_uso.md /guia_de_uso/index.html /guia_de_uso.html
      acercade/licencia.md /acercade/licencia/index.html /acercade/licencia.html

    • strict

      Determina el comportamiento del proceso de conversión tras producirse un aviso (warning). Por defecto es false. Hay que ponerlo como true para detener el proceso cuando se produce cualquier aviso.

    • dev_addr

      URL para mkdocs serve. Por defecto es '127.0.0.1:8000'. Debe escribirse en formato IP:PUERTO.

      También se puede establecer con el argumento --dev-addr al invocar mkdocs serve.

    Configurar extensiones

    MkDocs es un programa Python que traduce de markdown a HTML utilizando las funcionalidades proporcionadas por la biblioteca Python Markdown, que a su vez admite una gran variedad de extensiones para ampliar los recursos de sintaxis, como meta, toc, tables, o fenced_code.

    Por ejemplo, para utilizar SmartyPants, escribiremos:

    markdown_extensions:
    - smarty

    Algunas extensiones tiene opciones de configuración. Para asignarles un valor, añadir para cada una un mapa de elementos clave:valor.

    Por ejemplo, la extensión toc permite usar permalinks:

    markdown_extensions:
    - toc:
        permalink: True

    Véase que tras la extensión se ponen dos puntos y a continuación la lista de opciones:

    markdown_extensions:
    - toc:
        permalink: True
        separator: "_"

    Si la extensión utilizada no requiere opciones, basta con incluirla en la lista de extensiones:

    markdown_extensions:
    - smarty
    - toc:
        permalink: True
    - sane_lists

    Configurar complementos (plugins)

    Podemos aumentar las funcionalidades de MkDocs instalando complementos con el comando pip. De serie, MkDocs proporciona el complemento Search para realizar búsquedas, que es el único que se activa automáticamente.

    Para activar otros complementos, hay que especificar la lista en el archivo de configuración:

    plugins:
    - search
    - otros_complementos

    Para desactivar search, escribir una lista plugins que no incluya este complemento. El efecto será que desaparecerá el recuadro de búsqueda de la barra superior.

    Para asignar una lista vacía al parámetro plugins:

    plugins: []

    En grandes proyectos podemos mejorar el rendimiento de las búsquedas usando la opción prebuild_index del complemento search:

    plugins:
    - search:
        prebuild_index: true

    Por defecto es false.
